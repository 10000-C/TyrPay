# TyrPay

## 一句话介绍

TyrPay 是一个面向 Agent Economy 的可验证履约结算协议：先锁定付款，再用 zkTLS 证明 AI Agent 确实调用了约定的模型或 API，最后按结果自动放款或退款。

## 项目简介

AI Agent 正在成为新的生产力单元，但今天的 Agent 交易仍然缺少最关键的一层基础设施: **可信结算**。

买方无法确认卖方 Agent 是否真的调用了约定的模型、接口和参数；卖方完成服务后，也很难获得自动、可验证、可追溯的付款。结果是，Agent 与 Agent 之间的协作依然依赖人工信任、中心化平台仲裁，或者低效的事后扯皮。

TyrPay 解决的正是这个问题。

我们把一次 Agent 服务拆成三个可验证阶段:

1. 买方先发布任务并锁定资金。
2. 卖方提交执行承诺，明确将调用什么模型、什么 API、满足什么 usage 条件。
3. 卖方执行任务后提交 zkTLS 证明，验证服务端确实发生了约定调用；验证通过则自动放款，失败或超时则自动退款。

这意味着，TyrPay 不是“为结果买单”的模糊协议，而是“为可验证履约买单”的协议。它让 Agent 支付从“相信你做了”升级为“证明你做了”。

## 我们解决什么问题

- 解决 Agent 服务交付不可验证的问题。
- 解决 AI 调用过程黑箱、责任难界定的问题。
- 解决跨主体 Agent 协作中“先干活还是先付款”的信任死锁。
- 解决高价值 AI API / 模型调用缺少链上结算闭环的问题。

## 0G 集成

TyrPay 当前集成的核心 0G 组件是 **0G Storage**。

我们将 `Execution Commitment`、`Delivery Receipt`、`Proof Bundle`、`Verification Report` 等关键证明对象存入 0G Storage，链上只保存对象哈希与 URI。这样做有三个直接价值:

- 大体积证明数据不上链，降低结算成本。
- 证明材料可持久存储、可独立校验、可随时复查。
- 让一次 Agent 交易的承诺、执行、验证、结算形成完整的可验证证据链。

对 TyrPay 而言，0G Storage 不是“附件仓库”，而是整个可验证支付闭环中的 **证据可用层**。

## 核心机制

TyrPay 的最小闭环非常清晰:

- **链上结算层**：合约负责托管资金、记录任务状态、执行放款或退款。
- **承诺层**：卖方先声明可执行边界，避免事后篡改条件。
- **证明层**：zkTLS 为真实 API / 模型调用生成加密证明。
- **验证层**：Verifier 校验证明是否满足承诺。
- **存储层**：0G Storage 保存完整证明对象，保证可追溯与可审计。

一句话概括: **链上负责钱，zkTLS 负责真，0G 负责证据长期存在。**

## 为什么这个项目有价值

TyrPay 面向的是正在快速增长的 Agent-to-Agent 经济。未来会出现大量自动化采购、数据调用、模型推理、策略执行、内容生产与结果交付，但如果没有可信结算层，这些协作只能停留在 demo 或平台内闭环。

TyrPay 提供的，是 AI x Web3 世界里一块真正缺失的基础设施:

- 它让 AI 服务第一次可以被“证明已履约”。
- 它让支付不再依赖中心化平台背书。
- 它让 0G 上的 Agent 应用具备真实交易能力，而不只是展示能力。

## 结语

TyrPay 不是又一个支付界面，而是一层面向 AI Agent 协作时代的 **可验证结算协议**。

如果说 AI Agent 让机器开始独立工作，那么 TyrPay 要做的，就是让它们能够在陌生主体之间安全成交、可信履约、自动结算。

**让每一笔 Agent 服务，都有证据；让每一次付款，都以履约为前提。**
