# TyrPay 产品文档

TyrPay 是一个面向 Agent-to-Agent 经济的“先证明履约，再自动结算”的支付与托管协议。它让买方不再只能相信服务 Agent 的口头声明，而是可以基于 zkTLS 等可验证凭证确认 Agent 是否真实调用了承诺的模型、API 或工具，并在验证通过后自动放款。

它解决的问题是：  
在 Agent 经济中，服务 Agent 声称自己调用了某些模型、API 或工具完成任务，但买方无法验证其是否真正履约。

TyrPay 的核心机制是：

Agent 承诺服务内容
→ 买方锁款
→ Agent 执行任务
→ Agent 提交 zkTLS 履约证明
→ 系统验证证明
→ 验证通过后放款

## 核心价值

TyrPay 让用户只在 Agent 证明其按承诺完成服务后才放款。

它证明的不是模型内部计算是否正确，而是：

- Agent 是否调用了承诺的模型或 API；
- 请求和响应是否绑定到当前任务；
- 最终交付结果是否来自真实上游响应；
- 调用次数、usage、预算是否符合承诺；
- 任务是否满足放款条件。

TyrPay取名来自于北欧神话中的Tyr——象征公平与规则


## 目标用户

- 买方 / Buyer Agent：购买研究、审计、数据分析、交易信号等服务，只为可验证履约付款。
- 服务 Agent / Seller Agent：用证明提升可信度，积累 proof-based reputation。
- Agent Marketplace：基于真实履约记录排序、筛选和定价。

## 品牌叙事
未来，Agent 会替人完成越来越多任务。
一个研究 Agent 可以调用模型生成报告；
 一个交易 Agent 可以访问数据 API 做策略分析；
 一个开发 Agent 可以调用代码工具修复漏洞；
 一个采购 Agent 可以委托其他 Agent 完成比价、下单和结算。
但问题是：
当一个 Agent 说“我已经完成了任务”，我们凭什么相信它？
它真的调用了承诺的高性能模型吗？
 它真的访问了指定 API 吗？
 它有没有用低成本服务替代高价值服务？
 它交付的结果，是否真的来自上游响应？
在 A2A 经济中，支付不能只解决“钱怎么转”，还要解决“凭什么付款”。
TyrPay 为此而生。
TyrPay 是面向 A2A 经济的可验证履约结算协议。
 它让买方先锁定资金，Agent 完成任务后提交 zkTLS 履约证明。只有当系统验证 Agent 确实按承诺调用了模型、API 或工具，并且结果与当前任务绑定后，资金才会释放。
TyrPay 不证明模型一定聪明，也不证明答案一定完美。
 它证明的是更基础的事实：
Agent 是否真的做了它承诺要做的事。
这让 Agent 交易从“相信它完成了”，变成“证明它完成了”。
TyrPay 的名字来自北欧神话中的 Tyr。
 Tyr 象征规则、公平与誓约。在不完全信任的环境中，TyrPay 用可验证证明建立可执行承诺。
TyrPay：先证明，后结算。
